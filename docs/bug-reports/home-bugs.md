## BUG-HOME-001: Date filter params crash GET /transactions/public with 500

- **Severity:** HIGH
- **Endpoint / UI:** `GET /transactions/public?dateStart=...&dateEnd=...`
- **Test file:** `tests/api/home.spec.ts` (`test.skip` at line 55 — "supports dateStart and dateEnd query params")
- **Found:** 2026-06-13
- **Expected:** 200 with the public transactions feed filtered to the given date range, e.g. `GET /transactions/public?page=1&limit=10&dateStart=2020-01-01&dateEnd=2030-12-31` returns the matching results
- **Actual:** 500 server crash whenever `dateStart` and/or `dateEnd` query params are supplied
- **Impact:** Date-range filtering of the public feed is completely broken; any client passing these documented params gets a server error instead of filtered data
- **Suggested fix:** Validate and parse `dateStart`/`dateEnd` in the `GET /transactions/public` handler before building the query; coerce to valid Date objects and return 422 JSON on malformed input instead of letting the query layer throw

## BUG-006 (carry-forward): WCAG 2.1 AA violations on app pages (link-name rule)

- **Severity:** MEDIUM
- **Endpoint / UI:** Home page `/` (and all authenticated app pages sharing the top bar / sidebar)
- **Test file:** `tests/ui/home.spec.ts` (`test.skip` at line 228 — "no critical axe-core WCAG 2.1 AA violations on homepage")
- **Found:** 2026-06-12 (originally on `/signin`), confirmed on home `/` 2026-06-13
- **Expected:** All links have a discernible accessible name (axe-core `link-name` rule passes)
- **Actual:** axe-core reports `link-name` violations — icon-only links (top-bar/sidebar icons) have no accessible name. This is the same root cause as the signin-page BUG-006 and is shared app chrome.
- **Impact:** Screen reader users cannot determine the purpose of these links; fails WCAG 2.1 AA SC 4.1.2
- **Suggested fix:** Add `aria-label` attributes to all icon-only links in the shared app layout (top bar + sidebar)
