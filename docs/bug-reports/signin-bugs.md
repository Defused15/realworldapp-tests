## BUG-001: Missing required fields return 500 HTML instead of 422 JSON

- **Severity:** HIGH
- **Endpoint / UI:** `POST /users`
- **Test file:** `tests/api/signin.spec.ts`
- **Found:** 2026-06-12

## BUG-003: POST /login response exposes bcrypt password hash

- **Severity:** HIGH
- **Endpoint / UI:** `POST /login` (200 response)
- **Test file:** `tests/api/signin.spec.ts`
- **Found:** 2026-06-12
- **Expected:** `user` object in the response must NOT contain a `password` field
- **Actual:** `user.password` contains the full bcrypt hash (e.g. `$2a$10$...`)
- **Impact:** OWASP API3 Excessive Data Exposure; hash can be used in offline brute-force attacks
- **Suggested fix:** Strip the `password` field from the user object before serializing the login response

## BUG-005: Sign Up link click doesn't navigate due to Formik DOM re-render

- **Severity:** LOW
- **Endpoint / UI:** "Don't have an account? Sign Up" link on `/signin`
- **Test file:** `tests/ui/signin.spec.ts`
- **Found:** 2026-06-12
- **Expected:** Clicking the link navigates to `/signup`
- **Actual:** URL stays at `/signin`. Playwright's mouse-move triggers Formik `onBlur` validation on the Username field, causing a re-render that detaches the link's DOM node before the click event fires.
- **Impact:** Navigation from signin to signup is unreliable for automated and potentially real users
- **Suggested fix:** Debounce Formik's `onBlur` re-render, or use `mousedown` instead of `click` for navigation links so the event fires before re-render

## BUG-008: POST /login with remember=true sends Expires attribute instead of Max-Age

- **Severity:** LOW
- **Endpoint / UI:** `POST /login` (200 response, `remember: true`)
- **Test file:** `tests/api/signin.spec.ts` (Remember Me > Contract section)
- **Found:** 2026-06-14
- **Expected:** `Set-Cookie` header contains `Max-Age=2592000` (RFC 6265 compliant, 30 days in seconds) when `remember: true` is passed
- **Actual:** `Set-Cookie` header contains `Expires=<date 30 days out>` (legacy format) and NO `Max-Age` attribute. Session login (no remember) sends neither `Expires` nor `Max-Age`.
- **Impact:** Minor — `Expires` is widely supported but is legacy. `Max-Age` is preferred by RFC 6265 because it is unaffected by client clock skew. Not a security issue.
- **Suggested fix:** Configure the Express session middleware to set `maxAge` (in milliseconds) instead of / in addition to `expires` on the cookie when `remember: true`
