## Findings

### SEC-1 — `tmp` < 0.2.6 path traversal (HIGH) — FIXED

- **Vulnerability**: CVE-2026-44705
- **Package**: `tmp` (0.0.33 and 0.1.0), transitive via `@lhci/cli` → `inquirer` →
  `external-editor`, and `@lhci/cli` directly.
- **Severity**: HIGH (fix available in 0.2.6)
- **App bug or our dependency?**: Ours — a dev-only transitive dependency of
  Lighthouse CI. Not in any runtime/app path (this repo ships nothing).
- **Action**: Added an npm `overrides` entry forcing `tmp@^0.2.6`. After
  `npm install`, both copies resolve to `tmp@0.2.7`; `@lhci/cli` still loads
  (v0.14.0 verified). Re-scan: **no findings**.

## OWASP ZAP (DAST) — NOT RUN locally (environment)

The app runs as a **host process bound to `127.0.0.1`** (only PostgreSQL is in
Docker here). The ZAP baseline runs in a container and cannot reach a
host-loopback-bound app on Docker Desktop for macOS (`Connection refused` on
both `--network host` + `localhost` and `host.docker.internal`). The scan config
(`security/zap/rules.tsv`, `npm run security:zap`) is correct for CI on Linux,
where the app and ZAP share a reachable network. Tracked: run ZAP in the CI
`zap` gate once the app is reachable there (see BACKLOG — app-in-CI).
